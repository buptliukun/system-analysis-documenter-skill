# 高级系统分析模板使用指南

## 模板概述

本模板采用领域驱动设计（DDD）的思想，通过"用户用例 → 领域模型 → 技术实现"的思维路径来构建系统分析文档。

## 模板结构说明

### 第一部分：领域建模思维路径

模板的核心是领域建模思维路径：
- **用户用例分析** → **领域模型推导** → **技术实现**
- 通过识别参与者、梳理业务流程、提取业务规则等方式建立领域模型

### 第二部分：文档章节结构

1. **文档头部** - 包含项目名称、修订历史
2. **概述** - 包含术语表、需求背景、目标设定
3. **业务架构分析** - 用例分析、用户动线、业务用例图
4. **服务架构和领域模型** - 系统架构、技术选型、领域模型图
5. **业务边界分析** - 服务交互、对外接口
6. **数据与配置设计** - 数据表结构、ER图
7. **现有服务影响分析** - 影响评估
8. **数据分析** - 数据需求
9. **非功能性分析** - 风险、容量、限流、发布计划
10. **资源变更汇总** - 资源清单
11. **工作量与排期计划** - 任务分配、时间安排

## PlantUML 图表使用说明

模板充分利用PlantUML生成各类图表：

### 1. 领域模型图
```
@startuml
package "领域名" {
    class "实体名" as Entity {
        + 属性: 类型
        --
        + 方法(): 返回值
    }
}
@enduml
```

### 2. 系统架构图
```
@startuml
package "前端层" as Frontend {
    [组件A]
}
package "服务层" as Service {
    [服务A]
}
@enduml
```

### 3. 时序图
```
@startuml
actor "用户" as User
participant "服务A" as ServiceA
User -> ServiceA : 请求
activate ServiceA
ServiceA --> User : 响应
deactivate ServiceA
@enduml
```

## 领域建模方法

### 从用户用例推导领域模型
1. **识别参与者** - 系统涉及的用户/角色
2. **梳理业务流程** - 核心操作流程
3. **提取业务规则** - 约束和限制条件
4. **抽象业务概念** - 转换为领域实体
5. **建立领域关系** - 定义实体间关系

## 编写最佳实践

### 1. 重点突出
- 使用**加粗**标记关键指标和核心功能
- 使用删除线标记废弃功能：~~功能名~~

### 2. 结构化呈现
- 大量使用表格对比方案、评估工作量
- 使用层级列表组织信息

### 3. 可追溯性
- 记录文档修订历史
- 链接相关资源（MRD/PRD/视觉稿）

### 4. 风险控制
- 独立章节分析风险
- 详细容量和限流说明
- 应急预案制定

## 与其他文档的区别

相比传统系分文档，本模板：

1. **强调领域建模** - 从用例出发推导领域模型
2. **图形化表达** - 使用PlantUML自动生成图表
3. **结构化思维** - 从用例到领域再到实现的清晰路径
4. **完整性** - 覆盖从业务到技术实现的全部方面
5. **实用性** - 包含工作量评估和排期计划

## 适用场景

本模板特别适合：
- 复杂业务系统的架构设计
- 需要领域建模的项目
- 要求严格文档规范的场景
- 团队协作开发项目
- 需要向多方沟通技术方案的情况
